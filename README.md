
# hexagen-monaco

This project was generated by **HexaGen UI** and follows a **Modular Monolith** architecture using Yarn Workspaces and Turborepo.  
This structure gives you strong separation of concerns, fast local development, and a clear path to scale — either by adding more modules or extracting high-traffic ones into microservices later.

## 🚀 Getting Started

1. **Install dependencies**  
   ```bash
   yarn install
   ```

2. **Start development servers**  
   This runs the backend (api-gateway) and frontend (web) concurrently.  
   ```bash
   yarn dev
   ```

3. **Sync architecture changes**  
   Whenever you update `.architecture.yaml` (e.g. add a new module, entity, or use case), run:  
   ```bash
   yarn sync
   ```
   This generates the corresponding folder structure, boilerplate files, package.json, and tsconfig updates.

4. **Check architectural integrity**  
   Validate that your code still respects the boundaries defined in `.architecture.yaml`:  
   ```bash
   yarn lint:arch
   ```

```
│   │   │   ├── application/  # Use cases, ports (in/out), orchestration

├── tools/

```

## ✨ Manifest-First Workflow

This project encourages defining your architecture in `.architecture.yaml` **before** writing lots of code.  
The manifest acts as documentation + contract, and tooling (`sync` + `lint:arch`) keeps everything consistent.

### Example: Adding a "Notifications" Feature

1. **Update `.architecture.yaml`**  
   Add a new module under `modules`:

   ```yaml
   - name: notifications
     description: "Handles email and in-app notifications"
     entities:
       - Notification
       - NotificationTemplate
     use_cases:
       - SendNotification
       - ListUserNotifications
     infrastructure:
       external_apis:
         - sendgrid        # or resend, postmark, etc.
       messaging:
         driver: bull-mq   # optional queue for async delivery
   ```

2. **Sync the changes**  
   ```bash
   yarn sync
   ```

   This will:
   - Create `packages/notifications/`
   - Scaffold `domain/model/Notification.ts`, `application/use-cases/SendNotification.use-case.ts`, etc.
   - Generate `package.json` and `tsconfig.json`
   - Add path alias to `tsconfig.base.json` (e.g. `@hexagen/notifications`)

3. **Implement the details**  
   - Fill in the `Notification` entity logic
   - Write the use case business rules
   - Create the `SendGridAdapter` (or your chosen provider) in `infrastructure/external-apis/`
   - Expose the use case in `api-gateway` routes or another module

4. **Validate**  
   ```bash
   yarn lint:arch
   ```

   The linter checks:
   - No illegal cross-module imports
   - Declared entities/use-cases have corresponding files
   - Infrastructure adapters match declared drivers/providers

## Tips & Next Steps

- **Adding a new bounded context/module** → repeat the example above (copy-paste + rename is fine too)
- **Importing from another module** → only allowed via `@hexagen/shared` or published events
- **Going to production** → build with `yarn build`, deploy `apps/api-gateway` and `apps/web` separately or together
- **Extracting to microservice** → modules like `messaging` or `provenance` are already isolated and can be extracted with minimal changes

Questions or want to add support for a new provider / framework?  
The HexaGen UI can regenerate or extend this project anytime — just tweak your config and hit generate.
